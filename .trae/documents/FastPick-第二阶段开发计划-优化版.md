# FastPick 第二阶段开发计划（优化版）

## 项目概述
在已完成的 UI 框架基础上，实现图片加载、评级、预删除等核心功能。本方案优化了元数据处理策略，仅使用 WIC 原生 API，放弃 ExifTool 集成，降低复杂度。

---

## 一、核心决策回顾（优化后）

| 决策项 | 最终方案 | 理由 |
|--------|----------|------|
| 图片加载 | 一次性加载路径 + 虚拟滚动加载缩略图 | 路径扫描仅一次，虚拟滚动保证流畅 |
| 缩略图生成 | 系统缓存优先 → WIC 实时生成 | 速度快，内存占用低 |
| RAW 解码 | WIC 原生支持，提取内嵌预览 | 无需 ExifTool，降低复杂度 |
| 元数据写入 | **仅 WIC 处理 JPG Exif 评级** | 原生支持，无需外部依赖 |
| RAW 评级存储 | **系统文件缓存（NTFS Alternate Data Streams）** | 轻量方案，无需考虑其他软件兼容 |
| 数据持久化 | 无数据库，JPG 评级写入 Exif，RAW 评级写入 ADS | 轻量可靠 |
| 预删除显示 | 保留显示 + 红色角标 | 直观可见 |
| 多选评级 | 批量设置相同评级 | 符合摄影筛选习惯 |

---

## 二、技术方案调整说明

### 2.1 放弃 ExifTool 的理由
- **复杂度**：需要打包、分发、命令行调用，增加维护成本
- **依赖**：外部程序可能出现兼容性问题
- **本应用定位**：轻量级看图选图工具，无需考虑与其他摄影软件（Lightroom、Capture One 等）的评级互通

### 2.2 RAW 评级存储方案：NTFS Alternate Data Streams (ADS)
- **原理**：利用 NTFS 文件系统的备用数据流特性，将评级数据存储在文件本身的隐藏流中
- **优点**：
  - 无需修改 RAW 文件本身（保护原始数据）
  - 数据与文件绑定，移动/复制文件时 ADS 会跟随（同分区）
  - 轻量级，使用 Windows 原生 API 即可读写
- **限制**：
  - 仅在 NTFS 分区有效
  - 跨分区复制会丢失 ADS 数据
  - 上传云端通常会丢失 ADS
- **应对**：在应用内提供明确的提示，这是轻量级方案的已知限制

### 2.3 RAW 预览策略
- **WIC 支持**：优先使用 WIC 提取内嵌 JPG 预览
- **不支持格式**：显示占位符（带感叹号图标）
- **无内嵌预览**：显示占位符
- **不处理**：完整 RAW 解码（性能开销大，本应用定位为快速选图，非专业后期）

## 三、开发阶段划分

### 阶段 2.1：图片加载与解码（预计 1 周）

#### 任务 2.1.1：图片扫描服务
- **目标**：实现双路径图片扫描和路径管理
- **内容**：
  - 创建 `Services/ImageScanService.cs`
  - 扫描两个文件夹，获取所有 JPG/RAW 文件路径
  - 按文件名匹配 JPG/RAW，生成虚拟图片项列表
  - 路径失效检测和提示
- **交付物**：
  - `ImageScanService` 类
  - `PhotoItem` 模型类（文件名、JPG路径、RAW路径、评级、预删除标记）

#### 任务 2.1.2：缩略图生成服务
- **目标**：实现缩略图生成和缓存
- **内容**：
  - 创建 `Services/ThumbnailService.cs`
  - 优先读取 Windows 系统缩略图缓存（IShellItemImageFactory）
  - 系统缓存不存在时，用 WIC 异步生成（100×80）
  - 实现 LRU 内存缓存（最多 200 张）
- **交付物**：
  - `ThumbnailService` 类
  - 缩略图缓存管理器

#### 任务 2.1.3：虚拟滚动实现
- **目标**：实现缩略图区虚拟滚动
- **内容**：
  - 使用 ItemsRepeater + UniformGridLayout
  - 仅渲染可视区域的缩略图
  - 滚动时动态加载新区域，卸载离开区域的图片
- **交付物**：
  - 虚拟滚动缩略图列表
  - 流畅的滚动体验（2000+ 张图片）

#### 任务 2.1.4：预览图加载
- **目标**：实现单图/多图预览
- **内容**：
  - JPG 优先加载原图
  - RAW 优先使用 WIC 提取内嵌 JPG 预览
  - 不支持格式或无内嵌预览时显示占位符
  - 切换图片时释放前一张预览图内存
  - 多选时平铺显示前 5 张
- **交付物**：
  - 预览图加载器
  - 内存释放机制

## 四、技术实现细节

### 4.1 核心类设计

```
Models/
├── PhotoItem.cs              # 虚拟图片项
├── FileTypeEnum.cs           # 文件类型枚举
└── DeleteOptionEnum.cs       # 删除选项枚举

Services/
├── ImageScanService.cs       # 图片扫描服务
├── ThumbnailService.cs       # 缩略图生成服务
├── JpgMetadataService.cs     # JPG 元数据服务（WIC）
├── RawMetadataService.cs     # RAW 元数据服务（ADS）
├── FileOperationService.cs   # 文件操作服务
└── SettingsService.cs        # 设置服务

ViewModels/
└── MainViewModel.cs          # 主视图模型
```

### 4.2 关键技术点

| 技术点 | 实现方案 |
|--------|----------|
| 虚拟滚动 | ItemsRepeater + UniformGridLayout |
| 缩略图缓存 | LRU 算法，最多 200 张 |
| RAW 预览 | WIC 提取内嵌 JPG，不支持则显示占位符 |
| JPG 评级写入 | WIC `BitmapMetadata` API |
| RAW 评级存储 | NTFS Alternate Data Streams |
| 删除到回收站 | Windows API `SHFileOperation` |
| 配置持久化 | appsettings.json + Windows 注册表 |

### 4.3 RAW 处理策略

```
RAW 文件加载流程:
1. 尝试使用 WIC 提取内嵌 JPG 预览
   ├── 成功 → 显示预览图
   └── 失败 → 显示占位符（带感叹号）

RAW 评级存储:
1. 检查文件所在分区是否为 NTFS
   ├── 是 → 写入 ADS (filename.raw:FastPickRating)
   └── 否 → 提示用户"评级数据可能无法保存"
```

### 4.4 内存控制策略

- 缩略图缓存：最多 200 张（约 6.4MB）
- 预览图卸载：切换时释放内存
- RAW 降级：优先使用内嵌预览（2-5MB）
- 内存监控：超过 200MB 时清空缩略图缓存

## 五、错误处理策略

| 错误场景 | 处理方案 |
|----------|----------|
| 图片损坏 | 显示占位图 + 提示，保留条目 |
| RAW 无内嵌预览 | 显示占位图（带感叹号） |
| RAW 格式不支持 | 显示占位图（带感叹号） |
| 路径失效 | 标题栏红色提示 + 手动刷新 |
| JPG 元数据写入失败 | Toast 提示 + 日志记录 |
| RAW ADS 写入失败（非NTFS） | 提示用户"评级数据可能无法保存" |
| 文件被占用 | 重试 2 次，仍失败则提示用户 |

## 八、与原版方案的差异

| 项目 | 原版方案 | 优化版方案 | 影响 |
|------|----------|------------|------|
| 元数据工具 | ExifTool | 仅 WIC | 降低复杂度，无需外部依赖 |
| RAW 评级存储 | XMP 文件 | NTFS ADS | 仅限 NTFS，不跨软件兼容 |
| RAW 预览 | ExifTool 降级 | 仅 WIC，不支持则占位符 | 部分 RAW 可能无法预览 |
| 安装包大小 | 包含 ExifTool (~5MB) | 仅应用本身 | 减小体积 |
| 维护成本 | 高（需维护 ExifTool 集成） | 低（纯原生 API） | 降低维护负担 |

## 九、风险与应对

| 风险 | 应对措施 |
|------|----------|
| RAW ADS 数据丢失（跨分区复制） | 在应用内明确提示用户这是轻量级方案的已知限制 |
| RAW 无内嵌预览 | 显示占位符，这是性能优先的设计决策 |
| WIC 不支持某些 RAW 格式 | 显示占位符，本应用定位为快速选图，非专业后期 |
| 内存占用过高 | LRU 缓存 + 内存监控 + 主动释放 |
| 虚拟滚动卡顿 | 优化 ItemsRepeater 配置，减少重绘 |
